  <files>components/context-menu/context-menu.vue, components/context-menu/context-menu-item.vue</files>
  <action>
Per user decision: public API (types.ts, context-menu.ts with showContextmenu overloads) is already generic — do NOT change those files.

In `context-menu.vue`, change the props type:
- `context?: any` → `context?: unknown`
- `items: ContextmenuItem<any>[]` → `items: ContextmenuItem<unknown>[]`

In `context-menu-item.vue`, change the props type:
- `item: ContextmenuItem<any>` → `item: ContextmenuItem<unknown>`
- `context?: any` → `context?: unknown`

This works because:
- `ContextmenuItem<unknown>` resolves `action` to `(context: unknown) => void`
- The `item.action(context)` call in context-menu-item.vue passes `unknown` to a function accepting `unknown` — type-safe
- Vue's `h()` function in context-menu.ts has loose prop typing, so passing `ContextmenuItem<SiteItemUI>[]` to a component expecting `ContextmenuItem<unknown>[]` works at the render boundary
- The type safety is enforced at the `showContextmenu` call site (already generic), not at the internal SFC boundary

Verify no other `any` remains in either file after the change.

  <files>components/select/select.vue, components/select/index.ts, components/setting-modal/setting-modal.vue</files>
  <action>
Refactor `select.vue` to be a generic component per user decisions:
- **Explicit generic**: use `<script setup lang="ts" generic="V extends string | number">` (covers all current usage patterns)
- **Flexible option shape**: add `labelKey` and `valueKey` props with defaults `'label'` and `'value'`
- **Breaking change**: remove the fixed `Option` interface entirely

New select.vue script section structure:

1. Add `generic="V extends string | number"` to the script tag
2. Remove the old `interface Option { label: string; value: any; disabled?: boolean }`
3. Define props:
   ```ts
   const {
     options = [],
     labelKey = 'label',
     valueKey = 'value',
     placeholder = '请选择',
     disabled = false,
     width
   } = defineProps<{
     options?: Record<string, unknown>[]
     labelKey?: string
     valueKey?: string
     placeholder?: string
     disabled?: boolean
     width?: string | number
   }>()
   ```
4. Change `defineModel<any>()` → `defineModel<V>()`
5. Change `defineEmits<{ change: [value: any] }>()` → `defineEmits<{ change: [value: V] }>()`
6. Add accessor helpers:
   ```ts
   const getLabel = (opt: Record<string, unknown>) => String(opt[labelKey] ?? '')
   const getValue = (opt: Record<string, unknown>) => opt[valueKey] as V
   const isDisabled = (opt: Record<string, unknown>) => Boolean(opt.disabled)
   ```
7. Update `selectedOption` computed to use `getValue`:
   ```ts
   const selectedOption = computed(() => options.find((opt) => getValue(opt) === modelValue.value))
   ```
8. Update `selectOption` to use helpers:
   ```ts
   const selectOption = (option: Record<string, unknown>) => {
     if (isDisabled(option)) return
     const val = getValue(option)
     modelValue.value = val
     emits('change', val)
     isOpen.value = false
   }
   ```
9. Update template bindings:
   - `{{ selectedOption.label }}` → `{{ getLabel(selectedOption) }}`
   - `{{ option.label }}` → `{{ getLabel(option) }}`
   - `:class="{ active: modelValue === option.value }"` → `:class="{ active: modelValue === getValue(option) }"`
   - `option.disabled` → `isDisabled(option)`
   - `:key="String(option.value)"` → `:key="String(getValue(option))"`

**Caller updates (setting-modal.vue):**
The two NSelect usages in setting-modal.vue bind to `formData.wallpaperProvider` (string) and `formData.wallpaperInterval` (number). Both use default label/value keys with `{ label, value }` option objects. The generic V will be inferred from v-model binding type — no template changes needed in setting-modal.vue since the default labelKey/valueKey match existing option shapes.

Verify setting-modal.vue compiles without errors after select changes.
